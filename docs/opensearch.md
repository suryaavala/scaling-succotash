# OpenSearch Operations & Query Guide

This guide explains how to interact with the underlying OpenSearch 2.11 cluster, detailing interactions from basic REST commands to advanced hybrid vector retrieval workflows used in the Scaling Succotash project.

## 1. The Basics

OpenSearch is a distributed search and analytics suite built for speed. Everything in OpenSearch is exposed via a REST API natively communicating over JSON.

### Checking Cluster Health
Ensure the locally orchestrated cluster is running cleanly:
```bash
curl -s -X GET "http://localhost:9200/_cluster/health" | jq
```
*A `status` of `green` or `yellow` indicates the node is actively accepting requests.*

### Viewing Indices
An **Index** is intuitively similar to a table in a relational database. To see all indices currently stored:
```bash
curl -s -X GET "http://localhost:9200/_cat/indices?v"
```

### Retrieving a Document
If you know the `_id` of a specific company, you can retrieve it directly:
```bash
curl -s -X GET "http://localhost:9200/companies/_doc/<document_id>" | jq
```

## 2. Core Concepts: The Mapping (Schema)

Unlike standard databases, OpenSearch automatically infers types, but we explicitly provide a **Mapping** ensuring optimal search behavior.

### Text vs Keyword
- `text`: Fields mapped as `text` (like `description` or `name`) are **tokenized**. "Software Engineering" becomes `["software", "engineering"]`. This allows for partial match searches natively utilizing BM25 (TF-IDF).
- `keyword`: Fields mapped as `keyword` (like `country.keyword`, `industry.keyword`) are **not tokenized**. "Software Engineering" stays perfectly intact as `"Software Engineering"`. This is strictly utilized for **Exact Filtering**, Aggregations, and sorting mathematically.

### Dense Vectors (Embeddings)
Our application injects a specific `knn_vector` field called `description_embedding`. This holds a 384-dimensional floating-point array generated by PyTorch, mapped via HNSW (Hierarchical Navigable Small World) graphs structurally clustering semantic meanings simultaneously.

To view the exact mapping applied in our cluster:
```bash
curl -s -X GET "http://localhost:9200/companies/_mapping" | jq
```

## 3. Advanced Querying

All search requests are structurally sent as `POST` configurations to the `_search` endpoint directly.

### A. Deterministic Filtering (Exact Keyword Match)
When a user selects filters from the Streamlit sidebar, we don't want partial matches. We want absolute constraints utilizing the `.keyword` subfields.

```bash
# Find 5 companies located exactly in the United States inside the software industry
curl -s -X POST "http://localhost:9200/companies/_search" -H 'Content-Type: application/json' -d'
{
  "size": 5,
  "query": {
    "bool": {
      "must": [
        { "term": { "country.keyword": "united states" } },
        { "term": { "industry.keyword": "software" } }
      ]
    }
  }
}' | jq
```
*Note: We use `term` queries tightly mapping against `.keyword` explicitly avoiding token analysis entirely.*

### B. Standard BM25 (Full-Text Search)
If a user statically types a company name or keyword description natively.

```bash
# Finds companies mentioning "machine learning" in their description natively
curl -s -X POST "http://localhost:9200/companies/_search" -H 'Content-Type: application/json' -d'
{
  "size": 3,
  "query": {
    "match": {
      "description": "machine learning"
    }
  }
}' | jq
```

### C. k-NN Semantic Vector Search
When the intelligent agent triggers a search for "A financial startup innovating in blockchain logistics", rigid keywords fail completely. The Inference Service structurally translates that sentence into a 384-d vector mathematically, and we ask OpenSearch to calculate **Cosine Similarity** mapping neighbors dynamically globally.

```bash
curl -s -X POST "http://localhost:9200/companies/_search" -H 'Content-Type: application/json' -d'
{
  "size": 5,
  "query": {
    "knn": {
      "description_embedding": {
        "vector": [0.012, -0.045, 0.123 /* ... 384 elements ... */],
        "k": 5
      }
    }
  }
}' | jq
```

### D. The Holy Grail: Hybrid Search (Pre-Filter + Vector)
In our production tier, we combine both natively mapping exact rigid filters *alongside* fuzzy vector logic dynamically.

```bash
curl -s -X POST "http://localhost:9200/companies/_search" -H 'Content-Type: application/json' -d'
{
  "size": 10,
  "query": {
    "bool": {
      "filter": [
        { "term": { "country.keyword": "united kingdom" } }
      ],
      "must": {
        "knn": {
          "description_embedding": {
            "vector": [0.012, -0.045, 0.123 /* ... 384 elements ... */],
            "k": 100
          }
        }
      }
    }
  }
}' | jq
```
**Why this is powerful:** The `filter` execution path uses inverted indices skipping expensive vector math universally, radically shrinking the data pool to only UK companies *before* structurally calculating the `knn` distances locally.

## 4. Teardown & Maintenance

If the OpenSearch index becomes corrupted or you wish to start fresh with a new ingestion cycle:

```bash
# Delete the entire index
curl -s -X DELETE "http://localhost:9200/companies"

# Verify mapping is deleted
curl -s -X GET "http://localhost:9200/companies/_mapping"
# Should return {"error":{"root_cause":[{"type":"index_not_found_exception"...}]}}
```
After deletion, the `/search/health` endpoint on the FastAPI Gateway will natively automatically recreate the index schema with the correct `knn_vector` configurations precisely before the next Celery pipeline ingestion executes.
